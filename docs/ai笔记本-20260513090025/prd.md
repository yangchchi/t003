## 1. 文档概述

### 1.1 文档背景
在信息爆炸的时代，学生、职场人士及研究者等泛知识工作者面临信息过载、记录效率低下、知识碎片化难以关联和检索的核心痛点。传统的笔记工具（如普通记事本、简单云笔记）仅解决了“记录”问题，但在“整理、内化、复用”环节严重依赖用户个人能力，效率瓶颈明显。

本项目旨在开发一款AI驱动的智能笔记应用，通过集成自然语言处理（NLP）、光学字符识别（OCR）、自动语音识别（ASR）等AI技术，将笔记工具从“被动记录容器”升级为“主动知识助手”，帮助用户一站式完成信息的捕获、结构化、关联与高效检索，从而系统性提升个人知识管理效率。

### 1.2 项目目标
**业务目标 (3个月内MVP上线后6个月)：**
- 用户规模：实现注册用户数达到10万，月活跃用户（MAU）达到3万。
- 用户留存：次月留存率不低于40%。
- 商业化探索：为后续高级功能（如更大存储空间、更强大的AI模型）付费转化奠定基础，收集至少1000名付费意向用户。

**产品目标 (MVP版本)：**
1.  **核心功能完成度**：100%交付“智能记录”、“内容组织”、“智能检索”、“多端同步”四大核心功能。
2.  **性能指标**：语音转文字准确率≥95%（安静环境），笔记检索响应时间<1秒，三端同步延迟≤2秒。
3.  **用户体验**：应用商店评分（iOS/Android）不低于4.5分，用户核心任务完成率（创建-检索一条笔记）达90%以上。

**用户价值：**
- **对学生**：课堂录音实时转文字并提炼重点，课后快速生成复习大纲。
- **对职场人士**：会议记录自动结构化，行动项智能识别，历史资料秒级语义查找。
- **对研究者**：文献图片快速提取文字并自动归类，跨笔记内容深度关联，激发研究灵感。

### 1.3 受众范围

- **研发团队（前端、后端、算法、移动端）**：作为开发、测试的基准依据。
- **测试团队**：根据本文档编写测试用例，进行功能与性能验证。
- **运营与市场团队**：了解产品核心价值与亮点，为上线推广做准备。

## 2. 业务说明

### 2.1 用户角色
| 角色 | 核心特征 | 核心诉求与痛点 |
| :--- | :--- | :--- |
| **学生 (主要用户)** | 高频记录，信息源多样（课件、板书、录音），有复习备考需求。 | 1. 课堂记录速度跟不上讲解；2. 课后整理笔记耗时耗力；3. 考前复习找不到重点。 |
| **职场人士 (主要用户)** | 会议多，需记录行动项，知识沉淀需求强，注重效率与专业性。 | 1. 会议纪要整理繁琐；2. 过往项目经验分散，难以复用；3. 快速查找历史决策依据。 |
| **研究者/学者 (次要用户)** | 处理大量文献，需要深度阅读、摘录、关联想法。 | 1. 文献管理杂乱；2. 不同文献间的观点难以横向关联；3. 个人思考碎片化，难以形成体系。 |
| **访客** | 未注册用户。 | 体验产品核心功能，决定是否注册。 |

### 2.2 业务流程
**核心业务流程：信息输入 -> AI处理 -> 组织检索 -> 多端同步**

**正常流程：**
1.  **用户登录/注册**：用户通过手机号或邮箱注册并登录。
2.  **创建笔记**：
    - 路径A（文字输入）：用户手动输入文字，系统实时进行错别字检查与基础排版建议。
    - 路径B（语音输入）：用户点击录音，系统实时转写为文字，并在结束后自动分段、标点。
    - 路径C（图片输入）：用户上传或拍摄图片，系统调用OCR提取图中文字，并插入到笔记中。
3.  **AI自动处理**：笔记保存时/保存后，后台AI服务异步执行：
    - **内容分析**：提取关键实体（如人名、项目名、时间）、主题。
    - **自动标签**：根据主题和实体生成1-3个标签（如“会议纪要”、“Q3规划”、“机器学习”）。
    - **大纲生成**：识别标题（# H1, ## H2等Markdown语法或通过语义判断）和列表，生成侧边导航大纲。
    - **关联推荐**：在笔记底部或侧边栏，推荐内容相关的历史笔记（基于主题/实体相似度）。
4.  **用户组织与检索**：
    - 用户可基于自动生成的标签、手动标签、笔记本进行二次组织。
    - 在搜索框输入自然语言问题（如“上周开会说了哪些关于预算的事情？”），系统返回语义相关的笔记列表。
5.  **多端同步**：用户在任一端的操作（增删改笔记），通过同步机制在2秒内同步到其他已登录设备。

**异常流程：**
1.  **网络异常**：
    - **创建/编辑时断网**：提示“网络连接已断开”，自动将笔记草稿保存在本地。网络恢复后，自动尝试同步并提示“已同步至云端”。
    - **同步冲突**：当同一条笔记在A、B设备同时离线修改后上线，系统检测到冲突，以“最后修改时间”为准的版本为主版本，另一版本自动备份为“冲突笔记-时间”，并提示用户手动处理。
2.  **AI处理失败**：
    - 语音/图片识别置信度过低时，在转写结果处高亮显示，并提示“此部分识别可信度较低，建议您手动核对”。
    - 自动标签/大纲生成失败时（如内容过短或无结构），则不显示此部分，不产生错误提示，保持界面简洁。

### 2.3 使用场景
**场景一：学生的课堂高效记录**
> 小王在听一场关于“机器学习”的讲座。他打开AI笔记本，点击录音按钮。应用实时将教授的讲解转为文字并显示在屏幕上。讲座结束后，笔记已自动按语义分段，并高亮显示了“监督学习”、“神经网络”等关键术语。小王一键点击“生成大纲”，便得到了本次讲座的清晰结构，他稍作修改便形成了一份优质的复习资料。

**场景二：职场人士的会议纪要整理**
> 李经理在项目复盘会议中，用AI笔记本快速记录要点。会议结束后，她上传了拍摄的白板照片，文字被自动提取插入。系统自动给笔记打上了“项目A”、“复盘”、“2024Q2”等标签，并在笔记末尾关联推荐了半年前的《项目A启动会纪要》。李经理利用语义搜索“上次客户提出的主要需求”，快速找到了相关记录，完善了复盘报告。

**场景三：研究者的文献管理**
> 张博士在阅读一篇PDF论文时，对其中重要的图表和结论段落进行截图。他将截图批量导入AI笔记本，OCR准确提取了所有文字。系统根据内容自动将其归类到“深度学习-注意力机制”的标签下。几周后，当张博士思考一个新模型时，通过搜索“如何解决长序列依赖问题”，系统将这篇笔记与其他几篇相关笔记一并呈现，帮助他建立了知识连接。

### 3.3 页面说明（核心页面）
**1. 首页/笔记列表页：**
- **布局**：左侧导航栏（笔记本树、标签列表），右侧主区域为笔记列表和搜索框。
- **交互**：
    - 点击左侧笔记本或标签，右侧列表筛选显示对应笔记。
    - 点击搜索框，可进行关键词或语义搜索。
    - 列表项显示笔记标题、预览片段、最后修改时间、所属笔记本/标签。
    - 点击“新建笔记”按钮，进入空白编辑页。

**2. 笔记编辑/详情页：**
- **布局**：三栏布局（可选）。左栏：大纲导航（可收起）；中栏：编辑器/阅读区；右栏：笔记属性（创建时间、标签、关联笔记推荐）。
- **交互**：
    - **编辑模式**：工具栏悬浮顶部，支持文本格式和插入语音/图片。内容实时保存。
    - **阅读模式**：无工具栏，内容只读。点击大纲可快速定位。
    - 标签区域：显示自动标签（灰色）和手动标签（蓝色），可点击删除或添加新标签。

### 3.4 权限说明
- **所有注册用户**：拥有所有核心笔记功能（增删改查）的完整权限，但仅限操作自己的数据。
- **访客**：仅可体验语音转写、OCR的Demo功能（限制次数/时长），无法保存笔记，无法使用同步和检索功能。
- **【假设】未来版本**：可考虑“笔记分享（只读/编辑）”权限，但不在MVP范围内。

## 4. 非功能性需求

### 4.1 性能需求
- **页面响应时间**：除首次加载外，所有主要页面（列表、编辑）加载时间 < 1.5秒。
- **核心操作响应**：
    - 语音转文字延迟：< 300毫秒。
    - OCR处理时间（单张图）：< 3秒。
    - 笔记本地保存：< 200毫秒。
- **检索性能**：语义搜索响应时间 < 1秒（在用户笔记量<10万条的条件下）。
- **同步性能**：网络良好时，跨端同步延迟 ≤ 2秒。
- **并发支持**：MVP版本需支持至少5000用户同时在线。

### 4.2 兼容性需求
- **Web端**：支持 Chrome 90+、Safari 14+、Edge 90+ 的最新两个版本。
- **iOS端**：支持 iOS 15.0 及以上版本，适配 iPhone 主流机型。
- **Android端**：支持 Android 10.0 及以上版本，适配主流品牌机型（分辨率兼容）。
- **【说明】** 对于更低版本的系统，不保证所有功能可用，但应用应能正常安装和启动，并给出友好提示。

### 4.3 安全性需求
- **数据传输**：所有客户端与服务器通信必须使用 HTTPS (TLS 1.2+)。
- **数据存储**：
    - **云端**：用户笔记内容在服务器数据库加密存储（AES-256）。
    - **本地**：敏感数据（如笔记缓存、登录令牌）在移动端使用系统提供的安全存储（如iOS Keychain， Android Keystore）或进行等效加密。
- **认证与授权**：使用Token（JWT）进行用户认证，Token有过期和刷新机制。
- **隐私合规**：明确隐私政策，用户语音、图片数据仅用于当次转写/识别，完成后立即从服务器内存清除，不持久化存储原始音/图文件 **【关键约束】**。

### 4.4 可用性需求
- **系统可用性**：核心服务（笔记增删改查、同步）月度可用性不低于99.5%。
- **容错能力**：前端应用应有良好的错误捕获和提示，避免白屏崩溃。网络请求失败应有明确提示和重试机制。
- **离线可用性**：支持在无网络情况下查看、编辑、删除本地已有笔记。新建笔记可保存为本地草稿。
- **异常提示**：所有用户操作反馈（成功、失败、等待）需有明确、友好的非模态或模态提示。
